# Architecture

Kastel Stack is organised as an adaptive organism OS for businesses.

The public architecture is not a central command system. It is a protocol and scaffold for local business nodes that preserve their own DNA, sense their ecology, route decisions through human gates, recover from shocks and share only validated, contextualised learning with a maintained commons.

## Organism Layers

| Layer | Architecture responsibility | Primary artefacts |
| --- | --- | --- |
| DNA | Preserve the stable UVP, mission, protocol, claim boundaries and non-negotiables | DNA card, business scaling model hypothesis, claim-boundary record |
| Metabolism | Keep the business alive through weekly maintenance and slow scaling | weekly review, vitals, recurring maintenance checklist |
| Sensors | Gather internal and external signals | source-of-truth registry, manual observations, connector events |
| Nervous system | Interpret signals through semantic and causal structure | semantic model, ontology, knowledge graph, causal graph, inference kernel |
| Executive function | Route decisions through judgement and gates | decision cockpit, Action Intents, Clarify Intents, policy gates |
| Effectors | Make bounded changes in the world | content, outreach, CRM action, product change, partner move, operational fix |
| Immune system | Prevent collapse, capture, compliance failure and data loss | risk register, recovery checklist, continuity plan, public/private boundary |
| Learning system | Convert adaptation into evidence | delayed reviews, wrapper tests, source-of-truth outcomes, evaluations |
| Memory | Bank only portable operating intelligence | script bank, validation metadata, recovery protocols, invariants |
| Ecology | Model local fields and external niche conditions | adaptive field map, state-niche fit, platform/partner/regulatory context |
| Network | Share redacted learning without centralising control | commons artefacts, validation badges, node templates, hub-node stewardship |

## Node Tiers

### Light Node

The Light Node is the first adoption surface. It can run as a simple app, card system, spreadsheet or Markdown kit.

Required capabilities:

- DNA card
- weekly vitals
- robustness prior
- scaling prior
- adaptive field card
- decision card
- delayed review card
- script card
- recovery card
- shareability card
- exportable records

The Light Node hides ontology and causal-model complexity behind weekly prompts and simple validation badges.

### Managed Node

The Managed Node adds support from a hub, consultant, partner or trained operator.

Additional capabilities:

- guided weekly review
- domain starter packs
- simple CRM, payment or analytics integrations
- redacted outcome sharing
- proof tracking
- basic risk radar
- validation support

### Full Node

The Full Node implements the richer architecture.

Additional capabilities:

- semantic model
- domain ontology
- knowledge graph
- causal or generative model
- bounded AI helper agents
- multi-model routing
- source-of-truth exports
- AI-output evaluation
- business-outcome evaluation
- script-banking workflow
- security and recovery protocols
- network-sharing workflow

### Hub Node

The Hub Node is a mature Full Node that maintains shared infrastructure.

Responsibilities:

- Light Node onboarding
- script library maintenance
- field intelligence summaries
- risk radar updates
- proof-standard templates
- redacted learning bundles
- ontology primitives
- recovery-protocol libraries
- training and workshops
- validation curation
- commons maintenance economy

Hub Nodes steward the commons; they do not own local business truth.

## Event Fabric

Kastel's event fabric converts signals into governed learning:

```text
raw event
-> normalised signal
-> semantic object
-> field signal
-> causal update
-> decision card
-> Action Intent or Clarify Intent
-> observed outcome
-> delayed review
-> script-banking decision
-> optional network-sharing decision
```

Each transition should preserve context, source-of-truth mapping, confidence, risk and review status.

## Decision Routing

Decision routing uses the executive function of the organism.

Candidate moves are routed as:

- **Auto**: safe, reversible, low-risk and policy-approved.
- **Draft**: AI or workflow may prepare an artefact, but human review is required.
- **Escalate**: requires senior, legal, financial, clinical, privacy or strategic judgement.
- **Blocked**: unsafe, unclear, non-compliant, not source-of-truth grounded or outside current capacity.

When the niche is too opaque for useful action, Kastel emits a Clarify Intent instead of forcing an Action Intent.

## Immune System

The immune system protects the organism from:

- false claims
- privacy leakage
- security failure
- compliance drift
- operator overload
- cashflow shocks
- platform dependency
- model dependency
- unsupervised automation
- brittle single points of failure

Every important component should have a shark-tooth continuity answer: export, replacement, manual fallback, backup or degraded-mode operation.

## Commons Boundary

Network sharing is opt-in, redacted and metadata-rich.

The commons may store:

- validated scripts
- risk warnings
- proof patterns
- recovery protocols
- field intelligence
- localisation lessons
- claim-boundary refinements
- semantic definitions
- evaluation patterns

The commons must not require customer data, private business strategy, financial details, partner secrets or local autonomy.

The commons is useful only if it is maintained. Hub Nodes and paid node tiers should support a durable maintenance economy.
