# Weekly Priority Engine

The weekly priority engine is Kastel's metabolic rhythm.

It ranks maintenance actions, learning actions, Action Intents, Clarify Intents, delayed reviews, recovery steps and boundary checks using leverage, risk, evidence, dependency and workload signals.

## Weekly Light Node Loop

```text
check vitals
-> protect the core offer
-> spot environmental changes
-> choose one maintenance action
-> choose one learning action
-> review one previous result
-> update script status
-> decide whether anything can be shared
```

## Full Review Questions

The weekly review should ask:

- What must be maintained?
- What field changed?
- What opportunity or risk is most real?
- What action would teach most?
- What should be clarified first?
- What must be protected?
- What can be safely automated?
- What requires human judgement?
- What model route is appropriate?
- What result needs delayed review?
- What, if anything, can be banked?
- What validation badge applies?
- What must be backed up or made replaceable?
- What, if anything, can be safely shared with the network?

## Ranking Model

Candidate priorities should be ranked by:

```text
impact
+ information gain
+ proof value
+ state-niche fit
+ urgency
- workload
- risk
- dependency burden
```

The engine should prefer a bounded list of three to five weekly priorities.

## Priority Types

- **Maintenance action**: preserves metabolism, reliability or delivery quality.
- **Learning action**: tests a bounded hypothesis in a defined adaptive field.
- **Clarify Intent**: reduces uncertainty before action.
- **Action Intent**: proposes a gated effect on the business or market.
- **Delayed review**: checks whether a prior result still holds.
- **Recovery action**: repairs or documents a failure mode.
- **Shareability decision**: decides whether learning can be redacted and sent to the commons.

## Output Contract

Each weekly priority should include:

- priority type
- linked DNA / UVP
- adaptive field
- hypothesis
- expected learning
- source-of-truth outcome
- state-niche fit estimate
- risk level
- approval gate
- owner
- workload estimate
- rollback or stop rule
- delayed review date
- script-banking expectation
- shareability level

Without weekly review, Kastel risks becoming a task generator.

Without delayed review, it risks banking immediate wins as permanent truth.

Without recovery checks, it risks becoming adaptive but brittle.
