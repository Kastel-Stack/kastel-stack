
I would structure it around a **weekly Light Node rhythm** plus a **monthly strategic positioning review**.

The key shift is that Kastel is no longer just asking “what tasks should I do?” It is asking: **which model needs to change, whose model needs to change, what evidence would justify that update, and what bounded action would test it?** That fits the updated framing of Kastel as a supervised system for turning signals into model updates, bounded action, evidence, reusable scripts and durable operating intelligence. 

## Weekly schedule: Power-Positioning Light Node

The weekly aim should be:

> one maintenance action, one model-shaping action, one delayed review, one script update.

That maps directly onto the Light Node rhythm: check vitals, identify model mismatch, identify risk, identify opportunity, choose a maintenance action, choose a model-shaping action, review one previous result, and update script status. 

### Monday — Field sensing and vitals

**Task:** review the current field.

Check:

| Area            | Question                                                                                 |
| --------------- | ---------------------------------------------------------------------------------------- |
| Business vitals | What changed in sales, sign-ups, engagement, replies, churn, usage, support or cashflow? |
| Market/category | What language, concern, trend or opportunity is becoming more visible?                   |
| Customer model  | Are people understanding the offer correctly?                                            |
| Partner model   | Are potential partners seeing credibility, risk, mutual benefit and next steps?          |
| Founder model   | Am I seeing the opportunity field clearly, or am I stuck in noise/avoidance?             |
| Risk            | What could damage trust, claims, workload, money, delivery or reputation?                |

**Output:** a short `weekly-vitals.md` entry with 5–10 bullet points.

### Tuesday — Model mismatch and power-axis scan

**Task:** identify the highest-value model mismatch.

Ask:

```text
Whose model most needs updating this week?
```

Choose one primary target:

| Model           | Example mismatch                                           |
| --------------- | ---------------------------------------------------------- |
| Customer        | “This is just another brain-training app.”                 |
| Partner         | “Interesting, but too risky/unproven.”                     |
| Market/category | “IQ Coach is hard to classify.”                            |
| Institution     | “The evidence/ethics/claims boundary is unclear.”          |
| Self-model      | “I am not sure which niche has the steepest payoff curve.” |
| Operating model | “The business depends too much on founder memory.”         |

Then run a quick eight-axis scan:

| Axis                 | Weekly check                                                                     |
| -------------------- | -------------------------------------------------------------------------------- |
| Self-positioning     | Are you appearing credible, distinctive and agentic?                             |
| Resource control     | What valuable asset, proof, data, tool or expertise are you making more legible? |
| Network architecture | Which person, group or institution should be closer to you?                      |
| Agenda/framing       | What variable should you put into public attention?                              |
| Influence/exchange   | What reciprocal value are you offering?                                          |
| Deployment           | What visible outcome will you produce?                                           |
| Organisation         | What process, CRM entry, file, dashboard or SOP needs strengthening?             |
| Opportunity          | What bounded high-upside move is available now?                                  |

This fits the updated power ontology, where power becomes inspectable through actors, resources, sponsors, gatekeepers, narratives, trust assets, routines, opportunity windows and power scripts. 

**Output:** one chosen model mismatch and one chosen power axis for the week.

### Wednesday — Write 3–5 Action / Clarify Intents

**Task:** convert the mismatch into 3–5 candidate moves.

Each candidate should be framed as either:

**Action Intent:** a bounded move that tries to update an external model.

Example:

```text
Mission:
Send a claim-safe Exam Resilience pilot note to one academic contact.

Model to shape:
Academic partner’s model of credibility, risk and mutual benefit.

Target model update:
From “interesting but speculative app”
to “bounded theoretically grounded pilot with clear claim limits”.

Source-of-truth outcome:
reply quality, meeting request, evidence request, specific objection or named next step.

Delayed review:
14 days.
```

**Clarify Intent:** a move that updates your own model before acting.

Example:

```text
Mission:
Rank 10 potential partner niches for IQ Coach / Exam Resilience.

Model to shape:
Founder’s model of where the first high-upside route is most likely.

Source-of-truth outcome:
ranked shortlist by trust route, pilot fit, evidence requirement, access and feedback quality.
```

The updated Kastel framework explicitly recommends this: every weekly priority should specify the model it is trying to shape, the source-of-truth outcome and the delayed review. 

**Output:** choose only 1–2 moves to execute this week.

### Thursday — Execute the model-shaping move

**Task:** do the highest-value move.

Good weekly moves for long-term strategic positioning would be:

| Move type          | Example                                                                                    |
| ------------------ | ------------------------------------------------------------------------------------------ |
| Proof move         | Create one short claim-safe proof note or outcome snapshot.                                |
| Category move      | Publish one Substack/LinkedIn post reframing “power” as adaptive capacity, not domination. |
| Partner move       | Send one precise pilot/collaboration message.                                              |
| Network move       | Ask one credible person for feedback, introduction or critique.                            |
| Offer move         | Improve one landing-page section so the customer model becomes clearer.                    |
| Institutional move | Clarify evidence, ethics, claims and pilot boundaries.                                     |
| System move        | Add one CRM field, follow-up rule, script, dashboard or SOP.                               |

**Rule:** do not chase output volume. Choose the move most likely to update the right model.

### Friday — Delayed review and script banking

**Task:** review one action from 7, 14 or 28 days ago.

Ask:

```text
Did this action actually update the target model?
Did it improve trust, optionality, proof, resource access or network position?
Did it create interpretable evidence?
Was the workload acceptable?
Could this become a reusable script?
Should it be banked, revised or retired?
```

This is essential because Kastel uses delayed validation: actions should not become scripts until they survive review. It also treats scaling as constrained by claim safety, operator capacity, trust, finance, delivery and recovery. 

**Output:** update:

```text
delayed-review.md
script-bank.md
risk-register.md
opportunity-field.md
```

## Monthly schedule: Strategic positioning review

The monthly review should be less tactical. Its job is to maintain long-term positioning, category clarity, power capacity and compounding advantage.

### Week 1 of each month — DNA and category audit

**Task:** protect the deep proposition while adapting the wrapper.

Ask:

```text
Is the core DNA still stable?
Which surface wrappers are working?
Which wrappers are distorting the offer?
Which claim boundaries need tightening?
Which category phrase is becoming more legible?
```

For IQ Mindware, the stable DNA might remain:

```text
evidence-led far-transfer training
+ cognitive performance readiness
+ human-led AI augmentation
+ claim-safe proof development
```

while wrappers can vary:

```text
AI-proof thinking skills
exam resilience
adaptive intelligence training
decision quality under pressure
cognitive sovereignty
```

This follows the Kastel rule: keep the deep proposition stable, adapt the surface wrapper, validate the coupling, and bank only what survives feedback. 

**Output:** update `business-dna.md` and `positioning-map.md`.

### Week 2 of each month — Power capacity audit

Score each axis from 1–5:

| Axis                 | Monthly strategic question                                           |
| -------------------- | -------------------------------------------------------------------- |
| Self-positioning     | Am I increasingly seen as a serious category-defining actor?         |
| Resource control     | What assets do I now control that others need?                       |
| Network architecture | Am I closer to the right partners, sponsors, buyers or institutions? |
| Agenda/framing       | Am I shaping the terms of debate, or just reacting?                  |
| Influence/exchange   | Are people moving towards collaboration, trust or adoption?          |
| Power deployment     | Did I convert capacity into visible outcomes?                        |
| Organisation         | Is the business less dependent on memory, mood and manual follow-up? |
| Opportunity          | Did I act on any bounded high-upside openings?                       |

**Output:** choose one power axis to strengthen next month.

### Week 3 of each month — Proof, legitimacy and trust audit

**Task:** consolidate legitimacy.

Review:

```text
proof assets
testimonials
usage data
pilot feedback
academic references
claim boundaries
case-study fragments
Substack authority pieces
partner responses
```

Then ask:

```text
What proof asset would most reduce trust friction next month?
What claim is currently too weak, too vague or too risky?
What evidence would make the offer easier to believe?
```

The updated Kastel view treats power partly as the capacity to convert outcomes into durable legitimacy. 

**Output:** create or improve one proof asset.

### Week 4 of each month — Opportunity and network positioning review

**Task:** look for convex opportunities.

Ask:

```text
Where is the payoff curve steepening?
What bounded first move could test it?
Would success open multiple downstream futures?
Can I recover if it fails?
Who is the smallest credible actor I should approach?
```

This fits the updated opportunity-capacity layer: Kastel distinguishes reckless risk from convex opportunity, where downside is bounded, upside is high, feedback is interpretable, recovery is possible and timing advantage exists. 

**Output:** one opportunity probe for the next month.

## The best monthly strategic positioning outputs

By the end of each month, you want five concrete artefacts:

```text
1. Updated positioning map
2. Updated power-axis scorecard
3. One improved proof/legitimacy asset
4. One banked, revised or retired power script
5. One bounded opportunity probe for next month
```

## My recommended operating rhythm

For you specifically, I would keep it light:

| Cadence   |       Time | Output                                                  |
| --------- | ---------: | ------------------------------------------------------- |
| Monday    |  30–45 min | Weekly vitals and model mismatch                        |
| Wednesday | 60–120 min | One model-shaping action                                |
| Friday    |  30–45 min | Delayed review and script-bank update                   |
| Monthly   |  2–3 hours | DNA, category, power-axis, proof and opportunity review |

The strategic principle is:

> Do not try to “be more powerful” every week. Build one more unit of legitimate, feedback-coupled power each week.

That means one clearer model, one stronger trust asset, one better-positioned relationship, one reduced dependency, one reusable script, one proof fragment, one better category frame, or one bounded opportunity probe.

Over a year, that compounds into strategic positioning rather than scattered activity.

## Supabase schema addition

Add these tables to the earlier schema.

```sql
create table power_axes (
  id uuid primary key default gen_random_uuid(),
  key text unique not null,
  title text not null,
  core_question text not null,
  sort_order int not null
);

create table power_features (
  id uuid primary key default gen_random_uuid(),
  axis_id uuid references power_axes(id) on delete cascade,
  key text not null,
  title text not null,
  diagnostic_question text not null,
  examples text,
  sort_order int not null
);

create table power_feature_scores (
  id uuid primary key default gen_random_uuid(),
  node_id uuid references nodes(id) on delete cascade,
  feature_id uuid references power_features(id) on delete cascade,
  review_id uuid references weekly_reviews(id) on delete set null,
  score int check (score between 0 and 5),
  confidence int check (confidence between 1 and 5),
  evidence_note text,
  risk_note text,
  next_action text,
  created_at timestamptz default now()
);
```

Then the axis score can be calculated from feature scores:

```text
Axis score = mean of feature scores
Weighted axis score = mean(feature score × confidence weighting)
```

For MVP, use the simple mean. Later, add weighting.

## Recommended score scale

Use a 0–5 scale:

| Score | Meaning                  |
| ----: | ------------------------ |
|     0 | Absent or unknown        |
|     1 | Weak / fragile           |
|     2 | Present but inconsistent |
|     3 | Functional               |
|     4 | Strong                   |
|     5 | Strategic advantage      |

Add a separate **confidence score**:

| Confidence | Meaning                  |
| ---------: | ------------------------ |
|          1 | Guess                    |
|          2 | Some observation         |
|          3 | Repeated pattern         |
|          4 | Strong evidence          |
|          5 | Source-of-truth evidence |

This prevents false precision. A feature can be scored highly but with low confidence.

## Most important design principle

Do not make the app feel like a personality test.

Make it feel like a **strategic cockpit**:

```text
What power capacity is underdeveloped?
What evidence supports that?
What move would strengthen it?
What risk must be bounded?
What result would tell us whether it worked?
What script can be banked if it works again?
```

That gives you a much stronger MVP than a simple eight-score dashboard. It turns the power model into an operational system for building legitimate, feedback-coupled strategic position.
